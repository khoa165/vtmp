### CHOOSE TECH STACKS

Initially, we sketched the UML diagram for our project with blocks to represent the front end and the backend based on the list of our features

Before choosing our tech stacks, we were given a chance to participate in useful workshops about ReactJS, and HTML/CSS/JavaScript hosted by Khoa to get hands-on experience about web development.

Then, our team brainstormed and decided to list out all the stacks possible and we chose to follow the PERN stack, including PostgreSQL, Express.js, ReactJS, and Node.js.

We opted for this particular tech stack due to our team's extensive familiarity with SQL, thus ensuring a seamless transition to PostgreSQL. The choice of React and Express is not only popular but also benefits from comprehensive full-ecosystem backing and most of the mentors are well-versed with this stack.

Thanh Nguyen has expressed an interest in the backend, therefore he was responsible for implementing API endpoints

Later on, near the end of the project, after most of our features had been implemented successfully, we decided to mingle: the front-end team now switched to the backend and vice versa. We switch mostly to get all members to have exposure in both the front-end and back-end.

## PROJECT (2.5 month)

### SETUP PROJECT IN GITHUB AND VISUAL STUDIO CODE

#### GITHUB

- We used GitHub for version control throughout our project. Each developer was assigned dedicated issues. They proceeded to name their branch according to the issue they were working on their branches (e.g., trang-navbar). Khoa and Quan reviewed our code so the code is as clean and as scalable as possible. Following a comprehensive review, the code was integrated into the development branch.
- Our effort was directed towards maintaining the developed branch as the pinnacle of stability among working branches. In instances where a push to this branch encountered errors and bugs during testing, our team undertook the necessary measures to rectify the situation.

#### VS CODE

- In our VS Code environment, we initiated the project by creating a well-organized folder structure for both the client and server sides. To speed up coding, we utilized Copilot which greatly boosted code generation and made our development process more efficient.
- In addition to Copilot, we integrated the Prettier extension into our VS Code setup. This step was crucial to ensure consistent code formatting across the entire project.

### IMPLEMENTATION

#### FRONTEND

- We decided to make use of React framework for our front-end development
- Thanks to the React workshop hosted by anh Khoa, we, inexperienced developers, knew the general set up of React to work with, such as using Bootstrap, Reactstrap, etc.
- Later on, we discovered Material UI, which is our main library for styling our web more conveniently compared to styling with css for all the components.
- For the charts in the dashboard page, we utilize this library https://recharts.org/en-US/, however, thereâ€™s a downside for this library is that they donâ€™t have the â€˜titleâ€™ prop to go with the chart.
- Transactions page: we see the need of list view (option of view as table and view as visualization) hence we implement them.
- For the transaction card in Grid view, we intend to let users store the receipt virtually, hence by making each transaction hold an image of the receipt, users donâ€™t have to worry about losing their receipt.
- We made use of Firebase Authentication to verify usersâ€™ credentials. We decided to use this because it saved more time than actually coding the logic to verify the usersâ€™ credentials under limited time. When calling endpoints to sign up/in, it will save the usersâ€™ data to the database and will be in charge of verifying whether that credential is valid or not and return for us userID or error if invalid.
- Beside Firebase Authentication, we used Firebase Storage to store usersâ€™ images. When users upload their images, it will be saved in the Storage and return for us an URL link to that image, we will take that URL and store it in our database, which will have a field to capture the image URL, that is where we store the URL in that. Therefore, whenever we get a transaction, it will come with an URL directing us to the image stored in Storage, so that we can display it on screen.
- Context API is used to manage the state more efficiently since we didnâ€™t have time to learn Redux.
- Chatbot (UI): This is one of our core features, this chatbot can parse user input and return insight about transactions, categories, bucket and monthly transaction. It utilizes the natural language processing model from OpenAI and our little trained model.

#### BACKEND:

- Firstly, we had two options to build the database: MongoDb and PostgreSQL. After considering the convenience of implementing the project and the opportunity to learn new tech stacks, we decided to use PostgreSQL and Prisma Client. These were new tools for us, so it was a chance to implement and learn.
- To start setting up the database for the project, we created a PostgreSQL Database with Prisma's Data Platform. Prisma provides an Object-Relational Mapping (ORM) layer that allows us to interact with the database using programming language constructs and concepts, rather than using plain SQL. Once this setup was completed, all the logic for interacting with the database was written using Prisma's syntax. This helped us save a lot of time during the development of endpoints in the following steps.
- To fit within the project's three-month scope, we chose to write around 30 endpoints covering CRUD operations and essential calculations for wallets and transactions. To make implementation, debugging, and maintenance more convenient, we categorized all the endpoints into the following features: Transaction, Wallet, Filter, Calculation for transactions, and Authentication. We thoroughly tested all endpoints using Postman to ensure they function correctly, even with edge cases.
